# 拡張向けライトディスカバリー手順

## 目的
機能拡張に必要な既存システムと統合要件を迅速に分析する。

## 集中的なディスカバリー手順

### 1. 拡張ポイント分析
**統合アプローチの特定**:
- 既存の拡張ポイントやインターフェースを探す
- 変更範囲（ファイル、コンポーネント）を特定
- 既存パターンへの適合を確認
- 後方互換性要件を把握

### 2. 依存関係チェック
**互換性の検証**:
- 新規依存のバージョン互換性を確認
- API 契約の変更がないか検証
- パイプラインに破壊的変更がないことを確認

### 3. 簡易技術検証
**新規ライブラリのみ**:
- WebSearch で公式ドキュメントを確認
- 基本的な利用パターンを確認
- 既知の互換性問題を確認
- ライセンス互換性を確認
- 主要な発見を `research.md` に記録（技術整合セクション）

### 4. 統合リスク評価
**簡易リスクチェック**:
- 既存機能への影響
- 性能面の影響
- セキュリティ上の考慮
- テスト要件

## フルディスカバリーへ移行すべき条件
以下が見つかった場合、フルディスカバリーへ切り替える:
- 大きなアーキテクチャ変更が必要
- 複雑な外部サービス統合
- セキュリティに敏感な実装
- 性能クリティカルなコンポーネント
- 未知または資料の乏しい依存

## 出力要件
- 明確な統合アプローチ（`research.md` に境界影響を記録）
- 変更対象のファイル/コンポーネントの一覧
- 新規依存とバージョン
- 統合リスクと緩和策
- テストの重点領域
